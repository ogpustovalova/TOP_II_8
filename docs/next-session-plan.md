# План следующей сессии: воспроизводимость и сопровождение

## 1. Назначение

Этот документ предназначен для запуска новой рабочей сессии с чистым контекстом.
Цель сессии — подготовить воспроизводимое conda-окружение, автоматизировать
проверки репозитория, актуализировать техническую документацию и подтвердить
исполнение эталонных ноутбуков в чистом окружении.

Корень репозитория определяется командой `git rev-parse --show-toplevel`.

Базовая ветка: `urfu`.

Рабочая ветка: `urfu-next`.

Точка ответвления от `urfu`: `24f0672`.

Коммит является ориентиром, а не указанием выполнить reset. Фактическое состояние
Git при старте новой сессии имеет приоритет.

## 2. Базовое состояние

На контрольной точке подтверждено:

- ветка `urfu-next` создана от `urfu` на коммите `24f0672`;
- базовая ветка `urfu` синхронизирована с `origin/urfu`;
- до добавления этого документа рабочее дерево было чистым;
- выполнено 17 из 17 критериев `docs/quality-checklist.md`;
- 14 ноутбуков проходят `nbformat`;
- 167 code-ячеек проходят синтаксический разбор;
- все 88 code-ячеек эталонов исполнены без error outputs;
- 310 локальных Markdown-ссылок работают;
- 37 внешних ссылок ресурсного контура доступны;
- собственные материалы, ноутбуки и скрипты лицензированы по CC BY 4.0;
- сторонние ресурсы сохраняют собственные лицензии и ограничения.

## 3. Первый шаг новой сессии

До изменения файлов выполнить:

```bash
git rev-parse --show-toplevel
git branch --show-current
git status --short --branch --untracked-files=all
git rev-parse HEAD
git rev-parse urfu
git rev-parse origin/urfu
git rev-list --left-right --count urfu...HEAD
git log --oneline --decorate -10
```

Ожидаемое состояние контрольной точки:

```text
ветка: urfu-next
точка ответвления: 24f0672
urfu: 24f0672
origin/urfu: 24f0672
urfu...HEAD до первого коммита: 0 0
рабочее дерево: только ожидаемые изменения, описанные в этом плане
```

Если состояние отличается:

- ничего не сбрасывать и не удалять;
- не использовать `git reset --hard` и `git checkout --`;
- записать фактический HEAD;
- изучить новые коммиты через `git log urfu..HEAD`;
- изучить рабочий diff;
- определить, относятся ли найденные изменения к задачам этого плана;
- продолжить с фактической последней подтвержденной точки;
- не возвращать репозиторий к `24f0672` автоматически.

Если сам файл `docs/next-session-plan.md` еще не закоммичен, это ожидаемое
изменение. Его нельзя удалять как посторонний файл.

## 4. Система учета задач

Нужно одновременно использовать:

- todo-инструмент основной сессии;
- постоянный журнал `docs/maintenance-progress.md`.

В todo-инструменте должна находиться ровно одна основная задача в статусе
`in_progress`. Параллельная работа субагентов отражается одной агрегирующей
задачей, например `Проверить результаты параллельного пакета субагентов`.

В `docs/maintenance-progress.md` создать таблицу:

| ID | Задача | Исполнитель | Статус | task_id | Затронутые файлы | Проверка | Коммит |
|---|---|---|---|---|---|---|---|

Допустимые статусы:

- `pending` — работа не начиналась;
- `delegated` — субагент запущен, но результат не получен;
- `returned` — субагент вернул результат, но основная сессия его не проверила;
- `in_progress` — основная сессия выполняет задачу;
- `blocked` — зафиксирован блокер;
- `verified` — результат проверен основной сессией;
- `completed` — результат проверен и включен в итоговое состояние или коммит;
- `cancelled` — задача явно отменена с указанием причины.

Обязательные правила:

- запуск задачи не означает ее выполнение;
- запуск субагента не означает выполнение делегированной задачи;
- сообщение субагента `готово` не означает выполнение задачи;
- результат субагента сначала получает статус `returned`;
- основная сессия обязана изучить diff и запустить проверки;
- статус `verified` ставится только после проверки результата;
- статус `completed` ставится только после выполнения всех критериев приемки;
- пустой ответ или падение субагента не считается успехом;
- task ID каждого субагента записывается до продолжения другой работы;
- команды проверки и их результаты записываются сразу;
- журнал обновляется после каждого значимого этапа, а не в конце сессии.

## 5. Восстановление после падения или перезагрузки

После перезапуска основной сессии выполнить следующий алгоритм:

1. Проверить ветку, HEAD, рабочее дерево, точку ответвления от `urfu` и состояние
   базовой ветки относительно `origin/urfu`.
2. Прочитать `docs/maintenance-progress.md`.
3. Посмотреть коммиты после записанной контрольной точки.
4. Сопоставить измененные файлы с задачами и исполнителями в журнале.
5. Проверить наличие коммитов и доказательств для задач `completed`.
6. Для задач `verified`, но не закоммиченных, повторить короткую проверку приемки.
7. Для задач `delegated` попытаться продолжить субагента по сохраненному task ID.
8. Если субагента продолжить нельзя, изучить оставленный diff до запуска замены.
9. Не повторять дорогое обучение при совпадающих source-хэшах и успешном журнале.
10. Начать с первой задачи, зависимости которой выполнены, а результат не доказан.

Дополнительные правила восстановления:

- если журнал и Git противоречат друг другу, источником истины являются Git,
  фактические файлы и повторяемые проверки;
- если задача отмечена выполненной, но результат или проверка отсутствуют, задача
  считается невыполненной;
- если субагент успел изменить файлы, но не вернул ответ, изменения сначала
  изучаются, а не удаляются;
- не запускать второго субагента на ту же область, пока не проверен результат
  первого;
- не считать отсутствие сообщения об ошибке доказательством успеха.

## 7. Работа с субагентами

Допускается параллельно запустить три независимых направления.

### S1. Инвентаризация зависимостей

Задача:

- прочитать импорты во всех `tools/*.py`;
- прочитать импорты code-ячеек ноутбуков;
- сравнить зависимости с окружением `onn`;
- предложить минимальные conda- и pip-зависимости;
- ничего не редактировать.

Ожидаемый результат:

- перечень обязательных пакетов;
- предлагаемые версии;
- разделение пакетов между conda и pip;
- возможные конфликты;
- команды smoke-теста.

### S2. Автоматический валидатор

Разрешенная область:

- `tools/validate_repository.py`;
- отдельные тесты валидатора при необходимости.

Запрещенная область:

- notebooks;
- environment-файлы;
- отчеты и чек-листы.

Валидатор должен проверять:

- синтаксис Python-файлов;
- `nbformat` всех ноутбуков;
- синтаксис code-ячеек после исключения notebook magics;
- отсутствие error outputs;
- исполненность code-ячеек эталонов;
- существование локальных Markdown-ссылок;
- отсутствие подписанных URL, JWT и абсолютных локальных путей;
- число студенческих и эталонных notebooks;
- внешние ссылки только по отдельному флагу.

### S3. Документация сопровождения

Разрешенная область:

- `repository-tree.txt`;
- документация датасетов и кэшей;
- проект политики доступа к эталонам.

Запрещенная область:

- environment-файлы;
- валидатор;
- notebooks;
- вычислительные генераторы.

Для каждого субагента до запуска записать:

- task ID;
- разрешенные файлы;
- запрещенные файлы;
- критерии приемки;
- команды проверки.

После возврата субагента:

- не дублировать его работу;
- изучить созданные файлы;
- проверить отсутствие изменений вне назначенной области;
- выполнить команды проверки;
- при необходимости продолжить того же субагента по task ID;
- только после проверки поставить статус `verified`.

## 8. Подготовка conda-окружения

Основное имя окружения: `top-ii-8`.

Существующее окружение `onn` используется только как источник информации. Его
нельзя считать чистым и не следует клонировать целиком.

Известная рабочая конфигурация:

- Python 3.11.15;
- PyTorch 2.5.1+cu121;
- torchvision 0.20.1+cu121;
- Natasha 1.6.0;
- opencv-python 5.0.0.93;
- ultralytics 8.4.102.

Предварительный список дополнительных зависимостей:

- numpy;
- pandas;
- scikit-learn;
- matplotlib;
- nltk;
- pymorphy3;
- nbformat;
- nbconvert;
- nbclient;
- ipykernel;
- jupyter_core.

Окончательный список формируется только после инвентаризации импортов.

Для `environment.yml` использовать каналы:

- `pytorch`;
- `nvidia`;
- `conda-forge`.

Предпочтительное распределение на момент составления плана:

- Python и научный стек устанавливаются через conda;
- проверенные CUDA-wheels PyTorch 2.5.1 и torchvision 0.20.1 устанавливаются из
  официального индекса PyTorch, поскольку их conda-сборки CUDA 12.1 больше не
  разрешаются с актуальным conda-forge при создании окружения с нуля;
- отсутствующие или нестабильные в conda пакеты добавляются в секцию `pip`;
- глобальная конфигурация conda не изменяется.

Если окружение `top-ii-8` уже существует:

- не удалять его автоматически;
- проверить соответствие `environment.yml`;
- для чистой проверки использовать временное имя `top-ii-8-verify`.

Команды создания и проверки:

```bash
conda env create -f environment.yml
conda run -n top-ii-8 python -c "import torch; print(torch.__version__, torch.cuda.is_available())"
conda run -n top-ii-8 python -c "import torchvision, numpy, pandas, sklearn, matplotlib, nltk, pymorphy3, natasha, cv2, ultralytics, nbformat, nbconvert"
```

Kernel устанавливается только внутрь окружения:

```bash
conda run -n top-ii-8 python -m ipykernel install --sys-prefix --name top-ii-8
```

Не использовать исполняемый файл `jupyter` напрямую: ранее обнаруживался launcher
с невалидным shebang. Использовать:

```bash
conda run -n top-ii-8 python -m nbconvert ...
```

Критерии завершения подготовки окружения:

- окружение создается с нуля;
- все импорты проходят;
- `torch.cuda.is_available()` возвращает `True`, если GPU доступен;
- выводится ожидаемая модель GPU;
- статическая проверка репозитория проходит;
- conda не создает неожиданные tracked-файлы.

## 9. Автоматическая проверка

После реализации валидатора выполнить:

```bash
conda run -n top-ii-8 python tools/validate_repository.py
conda run -n top-ii-8 python tools/validate_repository.py --check-external
git diff --check
git status --short --branch --untracked-files=all
git fsck --no-progress
```

Сетевые ошибки анализируются отдельно. HTTP 403, rate limit или временный timeout
не равны отсутствующей ссылке.

Базовые показатели до добавления новых файлов:

- 15 Python-файлов;
- 14 notebooks;
- 167 code-ячеек;
- 88 исполненных code-ячеек эталонов;
- 0 error outputs;
- 310 локальных Markdown-ссылок.

После добавления файлов числа могут измениться. Валидатор не должен искусственно
подгоняться под старые числа. В отчет записываются фактические показатели.

- заявленную лицензию;

- SHA-256 при наличии;
- способ повторной загрузки;

Tracked notebooks нельзя перезаписывать до успешного завершения проверки.

- устройство;
- exit code;
- наличие error outputs;
- ключевую метрику;

Если notebook упал:

- не отмечать его выполненным;

- не запускать весь набор заново;
- после исправления перезапустить только затронутый notebook;
- при изменении общего генератора определить все затронутые notebooks.

Если source notebook и окружение не изменились, а журнал содержит успешный запуск
с совпадающим SHA-256, повторять дорогое обучение после перезапуска не нужно.

## 13. Контрольные точки Git

Перед коммитом выполнить:

```bash
git status --short --branch
git diff --check
git diff --stat
git diff
git log --oneline -10
```

В коммит не должны попадать:

- датасеты;
- кэши;
- веса моделей;
- временные outputs;
- секреты;
- абсолютные локальные пути.

Рекомендуемые логические коммиты:

```text
chore(env): добавить воспроизводимое conda-окружение
test: добавить автоматическую валидацию репозитория
docs: обновить сопровождение и техническую валидацию
```

Коммиты создаются только по явному разрешению пользователя в новой сессии.

Push выполняется только по отдельному явному указанию пользователя.

Новые коммиты нельзя помещать непосредственно в `urfu`. Первый разрешенный push
рабочей ветки выполняется командой `git push -u origin urfu-next`; последующие —
обычной командой `git push`.

После каждого коммита его hash записывается в `docs/maintenance-progress.md`.

## 14. Финальная проверка

Перед завершением убедиться:

- каждая задача журнала имеет статус `completed`, `blocked` или `cancelled`;
- ни одна задача не считается выполненной только по факту запуска;
- результаты всех субагентов проверены основной сессией;
- conda-окружение создается из `environment.yml`;
- smoke-тест импортов проходит;
- валидатор завершается с exit code 0;
- семь эталонов подтверждены в чистом окружении;
- `repository-tree.txt` актуален;
- документация кэшей заполнена;
- политика доступа к эталонам зафиксирована или явно ожидает решения;
- `docs/validation-report.md` содержит версии, команды и результаты;
- `docs/quality-checklist.md` остается согласованным;
- `git diff --check` проходит;
- рабочее дерево содержит только ожидаемые изменения;
- секреты, датасеты и кэши не добавлены.

- task ID субагентов;
- как проверены результаты каждого субагента;
- какие команды завершились успешно;
- какие notebooks были исполнены;
- какие метрики получены;
- какие задачи остались заблокированы;
- фактический HEAD;
- состояние рабочей ветки;
- выполнялись ли commit и push.

## 16. Главный принцип

Источниками истины являются Git, фактический diff, журнал с доказательствами и
результаты воспроизводимых проверок. Сам факт запуска задачи или субагента не
является доказательством выполнения.
