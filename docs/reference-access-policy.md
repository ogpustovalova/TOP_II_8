# Политика доступа к эталонным решениям

## Контекст

В репозитории студенческие задания (`kim-*.ipynb`) и эталонные решения
(`kim-*-solution.ipynb`) находятся рядом в директориях `M*/attachments/`. Ниже
описаны возможные варианты организации публикации без принятия окончательного
решения.

---

## Вариант 1: Публичный репозиторий со всеми материалами (текущий статус)

| Аспект | Описание |
|---|---|
| **Влияние на студентов** | Полный доступ к эталонам; могут сверяться, но и могут копировать без понимания |
| **Влияние на преподавателей** | Не требуется дополнительной инфраструктуры; все материалы в одном месте |
| **Технические шаги** | Ничего не менять |
| **Плюсы** | Нуль затрат на сопровождение; максимальная прозрачность; простота форка и рецензирования |
| **Минусы** | Эталоны доступны студентам 24/7; невозможность контролировать момент публикации решения; стимул к копированию |

---

## Вариант 2: Отдельная преподавательская ветка

| Аспект | Описание |
|---|---|
| **Влияние на студентов** | Основной точкой входа служит ветка `main` или `student-release` без эталонов, но в публичном репозитории студент может открыть и ветку `solutions` |
| **Влияние на преподавателей** | При обновлении эталонов нужно синхронизировать две ветки |
| **Технические шаги** | 1. Создать ветку `solutions` из текущего `main`.<br>2. В `main` удалить файлы `*-solution.ipynb` и заменить на заглушки.<br>3. Настроить CI, блокирующий публикацию solution-файлов в `main`.<br>4. Использовать защиту веток для управления изменениями, а не как средство конфиденциальности. |
| **Плюсы** | Простая организационная граница; один репозиторий; эталоны не видны в основной ветке |
| **Минусы** | Ветка публичного репозитория доступна всем; вариант не обеспечивает разграничение чтения и подходит только для организационного разделения |

---

## Вариант 3: Приватный репозиторий с эталонами

| Аспект | Описание |
|---|---|
| **Влияние на студентов** | Работают с публичным репозиторием без эталонов |
| **Влияние на преподавателей** | Нужно поддерживать два репозитория; при обновлении студенческих ноутбуков синхронизировать приватный |
| **Технические шаги** | 1. Создать приватный репозиторий `TOP_II_8-solutions`.<br>2. Перенести туда `*-solution.ipynb` и `tools/sol_m*.py`.<br>3. В публичном репозитории удалить solution-файлы и добавить ссылку на процедуру получения эталонов.<br>4. Добавить CI-проверку, предотвращающую публикацию эталонов в публичный репозиторий.<br>5. Настроить GitHub Actions для синхронизации несекретных изменений. |
| **Плюсы** | Полный контроль доступа; эталоны недоступны без приглашения |
| **Минусы** | Дополнительная инфраструктура; риск рассинхронизации; затраты на управление доступом; двойные CI-затраты |

---

## Вариант 4: Student release без `*-solution.ipynb`

| Аспект | Описание |
|---|---|
| **Влияние на студентов** | Получают чистовой срез репозитория (релиз или тег) без эталонов в начале семестра |
| **Влияние на преподавателей** | Основной репозиторий остаётся полным; перед семестром создаётся тег/релиз для студентов |
| **Технические шаги** | 1. Настроить скрипт сборки архива без solution-файлов.<br>2. Создать GitHub Release с подготовленным архивом.<br>3. В README разместить ссылку на актуальный student release.<br>4. Автоматизировать создание релиза через GitHub Actions. |
| **Плюсы** | Единый репозиторий для разработки; студенты работают с фиксированным срезом; эталоны не удаляются из `main` |
| **Минусы** | Нужно не забывать создавать релиз перед семестром; студенты могут заглянуть в `main` в любое время, если репозиторий публичный |

---

> **Примечание.** Окончательное решение о выборе варианта принимает пользователь (преподаватель/администратор репозитория). Данный файл фиксирует варианты для обсуждения и не является нормативным документом.
