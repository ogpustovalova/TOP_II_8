## 1. Сверка компетенций с КРМ v3.0 (критично)

**Проблема:** в РПД УрФУ используется собственная нумерация индикаторов DL-1.1–DL-1.6,
**конфликтующая с КРМ**. Например, в РПД «DL-1.2» = современные архитектуры,
а в КРМ «DL-1.2» = неглубокие сети (перцептроны, MLP). Это создаёт риск путаницы.

**Решение:** привести коды индикаторов в документах ФОС к нумерации КРМ v3.0.

## 2. Структура модулей (7 модулей — по разделам РПД)

**Принцип:** один модуль = один раздел РПД («Код раздела, темы») = одна
содержательная тема. Каждый модуль несёт свой набор КИУ и содержит один основной
КИМ; в модели измерения каждый КИУ представлен отдельной строкой.

| Папка | Модуль (раздел РПД) | КИУ (основные) |
|---|---|---|
| `M1-neuron-basics/` | **Модуль 1. Искусственные нейронные сети** | DL-1.1_Б, DL-1.2_Б |
| `M2-training/` | **Модуль 2. Обучение искусственных нейронных сетей** | DL-1.1_Б |
| `M3-dense-networks/` | **Модуль 3. Полносвязные нейронные сети** | DL-1.2_Б |
| `M4-cnn/` | **Модуль 4. Сверточные нейронные сети** | DL-1.3_Б, DL-1.4_Б, DL-1.12_Б, DL-3.1_Б |
| `M5-sequences/` | **Модуль 5. Нейронные сети для анализа последовательностей** | DL-1.5_Б, DL-4.1_Б |
| `M6-autoencoders/` | **Модуль 6. Автокодировщики** | DL-1.7_Б |
| `M7-optimizers/` | **Модуль 7. Алгоритмы обучения нейронных сетей** | DL-1.1_Б |
| `Project/` | Проектная работа (интегральный КИМ) | один или несколько согласованных КИУ уровня Б |
| `Exam/` | Зачёт / экзамен (промежуточная аттестация) | агрегация ранее подтвержденных КИУ уровня Б |

**Обоснование:** 7 модулей — это прямое отражение 7 разделов РПД УрФУ. Каждый модуль
формирует отдельный результат обучения и закрывается одним КИМ. Проект проверяет
интеграцию согласованных базовых КИУ, зачет — целостное освоение дисциплины.

## 6. Принципы, которым следовать (из методологии ITMO Top AI)

1. **Одна строка основного практического измерения → один КИУ** (КРМ). Один КИМ
   может содержать несколько строк, если каждое проверяемое действие и уровень
   указаны отдельно. К1/К2 и зачёт могут ссылаться на выборку ранее
   подтверждённых КИУ, но не создают новые практические свидетельства; проект
   подтверждает только КИУ, явно выбранные в согласованном плане.
2. **Дескриптор технологически нейтрален**, схема «действие — объект — условия».
   Технологический контекст (PyTorch/TensorFlow) — в варианте задания, не в дескрипторе.
3. **Backprop на NumPy перед фреймворком** — педагогический стержень (Аммар Али).
4. **Устная защита + вопросы «почему»** — основной контроль лабораторных и
   проекта в эпоху LLM. Письменные срезы отдельно проверяют теорию, расчёты и
   обоснование решений.
5. **Содержательные задания, а не механические:** показать переобучение, сравнить
   траектории, выбрать метрику под цену ошибок — а не «запустить модель с L2».
6. **Вес проекта соответствует заявленному результату.**
7. **Репозиторий — это ФОС** (не отдельный документ). Интерактивная таблица модели
   измерения в `README.md`.
8. **Прозрачность использования LLM:** фиксируются промпт, полученный результат
   и способ проверки; оцениваются корректность, воспроизводимость и защита.
9. **Целевые роли** берутся из моделей выпускника программ: ML Engineer и Data
   Analyst для АИИ; ML Engineer, Data Engineer и MLOps для ПИИ.
10. **Курс обновлять ежегодно** (концепции меняются каждый месяц).
